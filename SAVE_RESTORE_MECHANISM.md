# GBA RTS存档和恢复机制详解

## 1. 扇区概念

### 1.1 Flash存储布局

本项目使用512KB的Flash空间来保存完整的游戏状态，分为8个64KB的扇区：

```
总空间: 512KB (0x80000)
扇区大小: 64KB (0x10000)
扇区数量: 8个 (编号0-7)
```

### 1.2 扇区分配表

| 扇区 | 偏移地址 | 内容 | 大小 | 说明 |
|------|----------|------|------|------|
| 0 | 0x00000 | EWRAM (0x2000000-0x203FFFF) 第1部分 | 64KB | 外部工作RAM |
| 1 | 0x10000 | EWRAM 第2部分 | 64KB | 外部工作RAM |
| 2 | 0x20000 | EWRAM 第3部分 | 64KB | 外部工作RAM |
| 3 | 0x30000 | EWRAM 第4部分 | 64KB | 外部工作RAM |
| 4 | 0x40000 | IWRAM + 调色板 | 32KB + 1KB | 内部RAM和调色板 |
| 5 | 0x50000 | VRAM 前半部分 | 64KB | 显存前64KB |
| 6 | 0x60000 | VRAM后半 + OAM + IO + 系统状态 | 32KB + ... | 复合数据 |
| 7 | 0x70000 | 原始SRAM内容 | 64KB | 游戏存档数据 |

### 1.3 扇区6的详细布局

扇区6包含多种数据，布局如下：

| 偏移 | 大小 | 内容 | 说明 |
|------|------|------|------|
| 0x0000 | 0x8000 (32KB) | VRAM后半部分 (0x6010000-0x6017FFF) | 显存后32KB |
| 0x8000 | 0x0400 (1KB) | OAM (0x7000000-0x70003FF) | 精灵属性 |
| 0x8400 | 0x0080 (128字节) | spend_0x80内容 | 临时缓冲区 |
| 0x8500 | - | DMA1额外恢复数据 | 特殊处理 |
| 0x9000 | 0x0060 (96字节) | IO寄存器 (0x4000000-0x400005F) | 硬件寄存器 |
| 0x9060 | 0x0030 (48字节) | 音频寄存器备份 | spend_0x80+0x40的内容 |
| 0x9090 | 0x0370 | IO寄存器 (0x4000090-0x40003FE) | 更多硬件寄存器 |
| 0xFFF0 | 0x0010 (16字节) | RTS标志字符串 | "EZ-OmegaRTCFILE." |

## 2. 临时存储区(spend_0x80)

### 2.1 位置和作用

- 地址：0x0203FE00 (EWRAM末尾附近)
- 大小：128字节 (0x80)
- 作用：保存中断时的CPU状态和临时数据

### 2.2 内容布局

| 偏移 | 大小 | 内容 | 说明 |
|------|------|------|------|
| 0x00-0x1F | 32字节 | r4-r11 | IRQ模式下的通用寄存器 |
| 0x20-0x23 | 4字节 | sp | IRQ模式下的栈指针 |
| 0x24-0x27 | 4字节 | lr | IRQ模式下的链接寄存器 |
| 0x28-0x2B | 4字节 | SPSR | IRQ模式下的程序状态寄存器 |
| 0x2C-0x2F | 4字节 | 保留 | - |
| 0x30-0x33 | 4字节 | 系统模式SP | 游戏运行时的栈指针 |
| 0x34-0x37 | 4字节 | 系统模式LR | 游戏运行时的返回地址 |
| 0x38-0x3F | 8字节 | 保留 | - |
| 0x40-0x6F | 48字节 | 音频寄存器 | 0x4000060-0x400008F |
| 0x70-0x71 | 2字节 | 保留 | - |
| 0x72-0x79 | 8字节 | DMA控制寄存器 | DMA0-3的CNT_H |
| 0x7A-0x7F | 6字节 | 保留 | - |

## 3. 保存流程

### 3.1 保存顺序

1. **中断处理准备** (keypad_process)
   - 检测L+R+START组合键
   - 保存音频寄存器到spend_0x80+0x40
   - 保存DMA控制寄存器到spend_0x80+0x72
   - 禁用音频和DMA

2. **主保存流程** (rts_save)
   - 启用绿色交换标志
   - 识别Flash类型
   - 擦除整个512KB空间
   - 按以下顺序保存：

### 3.2 详细保存步骤

```
1. 保存原始SRAM到扇区7
   目的：保护游戏原始存档数据

2. 保存EWRAM (256KB) 到扇区0-3
   方法：分4次，每次复制64KB到SRAM，再写入Flash

3. 保存IWRAM (32KB) + 调色板 (1KB) 到扇区4
   IWRAM -> SRAM[0x0000]
   调色板 -> SRAM[0x8000]

4. 保存VRAM前64KB到扇区5
   直接复制0x6000000-0x600FFFF

5. 保存复合数据到扇区6
   - VRAM后32KB
   - OAM数据
   - 系统模式SP/LR（此时保存）
   - spend_0x80完整内容
   - IO寄存器（两段）
   - RTS标志字符串

6. 恢复原始SRAM
   从扇区7恢复，确保游戏存档不受影响
```

### 3.3 关键技术点

- **系统模式SP/LR的保存时机**：在save_vram_back_misc_to_flash中临时切换到系统模式获取
- **IO寄存器的选择性保存**：只保存必要的寄存器，避免保存只读或危险的寄存器

## 4. 恢复流程

### 4.1 恢复触发

检测L+R+SELECT组合键后：
1. 首先验证RTS标志（从扇区6的0xFFF0读取）
2. 如果标志匹配"EZ-OmegaRTCFILE."，继续恢复
3. 否则直接返回，避免恢复损坏的数据

### 4.2 恢复顺序

恢复顺序与保存顺序基本一致，但有些特殊处理：

```
1. 恢复EWRAM (扇区0-3 -> 0x2000000)
2. 恢复IWRAM和调色板 (扇区4)
3. 恢复VRAM (扇区5-6)
4. 恢复OAM
5. 选择性恢复IO寄存器（使用register_list）
6. 恢复DMA1额外数据
7. 恢复spend_0x80
8. 恢复系统模式SP/LR
9. 恢复原始SRAM (扇区7 -> 0xE000000)
10. 恢复IRQ模式寄存器并返回
```

### 4.3 IO寄存器恢复策略

使用预定义的寄存器列表(register_list)，只恢复安全的寄存器：
```c
const uint16_t io_register_list[] = {
    0x0000, 0x0002, 0x0004, // 显示控制
    0x0008, 0x000A, 0x000C, 0x000E, // 背景控制
    // ... 更多寄存器
    0xFF00  // 结束标记
};
```

## 5. 回到游戏的机制

### 5.1 中断处理流程

```
游戏运行 -> 按键中断 -> keypad_irq_handler
              |
              v
        保存r4-r11,sp,lr,SPSR到spend_0x80
              |
              v
        调用keypad_process
              |
              v
        存档/读档处理
              |
              v
        从spend_0x80恢复寄存器
              |
              v
        跳转到原始中断处理程序
```

### 5.2 时间点说明

- **保存时间点**：按下L+R+START的瞬间，游戏状态被完整保存
- **恢复时间点**：恢复后会回到之前保存时的确切状态，包括：
  - CPU所有寄存器状态
  - 内存内容（EWRAM/IWRAM/VRAM）
  - 硬件状态（IO寄存器、DMA、音频等）

### 5.3 无缝恢复的关键

1. **完整的状态保存**：包括所有影响游戏运行的数据
2. **正确的恢复顺序**：确保依赖关系正确
3. **寄存器状态维护**：通过spend_0x80保证中断前后状态一致
4. **原始中断链**：处理完成后继续原始的中断处理流程

## 6. 数据完整性保证

### 6.1 RTS标志验证

16字节的标志字符串"EZ-OmegaRTCFILE."用于：
- 验证存档数据有效性
- 防止恢复损坏或未初始化的数据
- 兼容EZODE的存档格式

### 6.2 原子性操作

虽然没有真正的事务支持，但通过以下方式尽量保证数据一致性：
- 先保存原始SRAM，最后恢复
- 按固定顺序执行操作
- 使用标志验证数据完整性

这种设计确保了即使在极端情况下（如存档过程中断电），至少游戏原始存档不会损坏。
