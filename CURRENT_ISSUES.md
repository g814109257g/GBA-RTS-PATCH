# 当前技术问题和调试记录

## 1. 核心问题：栈空间不足

### 1.1 问题描述

在`run_thumb_from_ram`函数中，需要将Flash操作函数复制到栈上执行。这些函数可能很大（如`program_flash_4`约1KB），而IRQ模式的栈空间有限。

### 1.2 问题表现

- 复制大函数到栈时可能导致栈溢出
- 表现为游戏崩溃或行为异常

### 1.3 当前解决方案

在`run_thumb_from_ram`中切换到系统模式：
```c
"mrs r0, CPSR\n"          // 备份当前CPSR
"mov r1, #0xDF\n"         // 切换到系统模式
"msr cpsr_cf, r1\n"
"nop\n"
"push {r0}\n"             // 保存CPSR到栈（注意：此时已经是系统栈）
```

这样可以使用系统模式的栈（通常更大），但引入了新的复杂性。

## 2. 模式切换的复杂性

### 2.1 多重模式切换

当前的调用链涉及多次模式切换：
```
IRQ模式 (中断处理) 
    -> 系统模式 (run_thumb_from_ram)
    -> IRQ模式 (返回)
```

### 2.2 潜在问题

1. **栈一致性**：必须确保在正确的模式下使用正确的栈
2. **寄存器保存**：不同模式有独立的寄存器组，需要正确保存/恢复
3. **CPSR管理**：需要正确保存和恢复程序状态寄存器

## 3. 尝试过的其他方案

### 3.1 方案A：汇编包装函数（已尝试但失败）

创建`rts_save_wrapper`在汇编中处理模式切换：
```asm
rts_save_wrapper:
    @ 保存寄存器
    @ 切换到系统模式
    @ 调用rts_save_core
    @ 切换回IRQ模式
    @ 恢复寄存器
```

**结果**：调用C函数时崩溃，即使是空函数

**可能原因**：
- 系统模式栈未初始化
- 编译器生成的代码有特殊假设
- 函数调用约定问题

### 3.2 方案B：完全在IRQ模式运行（当前方案）

保持在IRQ模式，只在`run_thumb_from_ram`内部切换模式。

**优点**：
- 减少了模式切换的复杂性
- 大部分代码在单一模式下运行

**缺点**：
- 仍然需要在关键位置切换模式
- 代码复杂度较高

### 3.3 方案C：使用固定的RAM缓冲区（未实施）

不使用栈，而是使用固定的RAM区域执行Flash函数。

**优点**：
- 避免栈空间限制
- 不需要模式切换

**缺点**：
- 需要找到可用的RAM区域
- 可能与游戏冲突

## 4. Flash操作的特殊性

### 4.1 不同Flash类型

项目支持4种Flash类型，每种有不同的操作序列：
- Type 1: 标准Intel命令集
- Type 2: AMD命令集变体1
- Type 3: AMD命令集变体2  
- Type 4: 特殊的1KB块编程

### 4.2 识别流程的复杂性

某些Flash类型的识别需要特殊处理：
```c
// Type 1和4的识别代码相似但返回值相反
if (data != 0x96) {
    return 0;  // Type 1返回0
    return 1;  // Type 4返回1
}
```

## 5. 调试困难

### 5.1 缺乏调试手段

- 无法使用printf或断点
- 只能通过视觉反馈（如绿色交换）判断执行状态
- 错误时通常直接崩溃，难以定位问题

### 5.2 尝试的调试方法

1. **绿色交换标志**：
   ```c
   volatile uint16_t *green_swap_reg = (volatile uint16_t*)0x04000002;
   *green_swap_reg = 1;  // 屏幕应该变绿
   ```

2. **强制空白**：
   ```c
   mov r3, #0x04000000
   ldr r2, [r3]
   orr r2, r2, #0x0400  // 设置强制空白位
   str r2, [r3]         // 屏幕应该变黑
   ```

3. **分段测试**：逐步注释代码，定位崩溃位置

## 6. 可能的改进方向

### 6.1 简化模式管理

考虑重新设计，减少模式切换：
- 在游戏初始化时设置更大的IRQ栈
- 或完全避免栈上执行，使用其他方法

### 6.2 更好的错误处理

添加错误检测和恢复机制：
- 检查栈指针范围
- 验证Flash操作结果
- 添加超时机制

### 6.3 优化Flash函数

减小Flash函数大小：
- 合并相似代码
- 使用更紧凑的指令序列
- 考虑分块执行

## 7. 需要外部帮助的问题

### 7.1 系统模式栈初始化

**问题**：不确定游戏是否正确初始化了系统模式的栈指针

**需要了解**：
- GBA游戏的标准启动流程
- 各模式栈的典型配置
- 如何安全地检查和设置栈指针

### 7.2 编译器行为

**问题**：不清楚编译器对函数调用的具体假设

**需要了解**：
- ARM EABI的具体要求
- GCC在不同优化级别的行为
- 如何生成真正的"naked"函数

### 7.3 Flash编程最佳实践

**问题**：当前的Flash操作可能不是最优的

**需要了解**：
- 各种Flash芯片的最佳编程序列
- 如何提高编程速度
- 错误处理和重试机制

## 8. 总结

当前的主要障碍是模式切换和栈管理的复杂性。虽然基本功能已经实现，但在某些边缘情况下仍可能出现问题。需要更深入地理解GBA的系统架构和编译器行为，才能找到更优雅的解决方案。
